# Roadmap Update Skill

You are helping update the project's **Feature Roadmap** - both the feature plans in `pods/*_Plan.md` and the consolidated view in `roadmap.md`.

> **Note**: This skill handles the Feature Roadmap (what features/boulders we're building, in which milestones, and when).
> Sprint-level task execution lives in **ClickUp**, not here.
> For Validation Roadmap updates (hypotheses, BHQs, SHQs), see the validation-review skill.

---

## Project Structure

### Source of Truth (authored by humans)
- `product_targets.md` - Milestone definitions, must-have features, success criteria (stable, rarely changes)
- `pods/*_Plan.md` - Per-pod feature priorities by milestone (updated by pod leads)
- `features/*.md` - Feature specs with estimates, discipline breakdown, implementation approach
- `capacity.md` - Team staffing by discipline and milestone

### Generated / Consolidated
- `roadmap.md` - **Consolidated feature list + Gantt chart** (regenerated by this skill from pod plans)

### Reference
- `dependency_map.md` - Pod and feature dependencies
- `GlobalRules.md` - Cross-project constraints
- `ValidationRoadmap.md` - Validation hypotheses and SHQs

---

## Your Task

### 1. Read Current State
- Read the relevant `pods/*_Plan.md` file(s) for the pod(s) being updated
- Read `product_targets.md` for milestone definitions and targets
- Read `roadmap.md` for the current consolidated view and Gantt
- Read `capacity.md` to check staffing feasibility
- Read `dependency_map.md` for constraints
- If cross-pod impact is possible, read all pod plans
- Read relevant `features/*.md` if estimates or approach are changing

### 2. Gather Updates
Ask the user for:
- Which pod(s) to update?
- Feature status changes (started, completed, blocked)?
- New features to add?
- Priority or timeline changes?
- Features to move between milestones?
- Estimate revisions?

### 3. Validate Changes
- **Check capacity**: Does `capacity.md` support the planned work? Flag overallocation.
- **Check dependencies**: Will changes break dependent pods or features?
- **Check targets**: Do changes affect must-have features in `product_targets.md`? Flag if a must-have is being cut or delayed.
- **Check validation**: Do changes affect SHQs in `ValidationRoadmap.md`? (mention if so, but don't update it here)
- Flag risks if you spot them

### 4. Update Pod Plan
- Update the relevant `pods/*_Plan.md` file(s) with new priority order, statuses, estimates
- Update "Last Updated" timestamps
- If a new feature is added, note that a `features/*.md` doc should be created (or create it)

### 5. Update Feature Docs (if needed)
- If estimates changed, update the feature doc's estimate and implementation flow
- If scope changed, update the feature doc
- If discipline needs changed, update the feature doc and flag in `capacity.md`

### 6. Regenerate Consolidated Roadmap
**Always regenerate `roadmap.md` after updating pod plans.** This means:

#### a. Rebuild the Consolidated Feature Tables
For each milestone, list ALL features from ALL pod plans in one table:

```markdown
| Pod | Feature | Estimate | Status | Feature Doc |
|-----|---------|----------|--------|-------------|
```

#### b. Rebuild the Gantt Chart
Follow the Visual Roadmap Rules below.

#### c. Add Target Check Note
After each milestone's table, note whether planned features cover the must-haves from `product_targets.md`:
- "Target check: All must-haves covered" or
- "Target check: MISSING [feature] from product_targets.md"

### 7. Summary
- Summarize what changed
- Highlight risks, dependency impacts, or target gaps
- If changes affect Validation Roadmap SHQs, note which ones
- Suggest next steps

## Visual Roadmap Rules

### Mermaid Gantt Syntax

```
gantt
    title [Chart Title]
    dateFormat YYYY-MM-DD
    axisFormat %b '%y
    tickInterval 4week

    section Milestones
    [Name]  :milestone, [id], [date], 0d

    section [Pod Name]
    [Feature Name]  :[status], [id], [start or dependency], [duration]
```

### Status Mapping

| Pod Plan Status | Gantt Status | Visual |
|----------------|-------------|--------|
| Completed | `done` | Gray bar |
| In Progress | `active` | Blue bar |
| Blocked | `crit` | Red bar |
| Not Started | (no tag) | Default bar |

### Task ID Rules

- Pod prefixes: `emp` (Empire), `meta` (Metagame), `bat` (Battle), `soc` (Social Dynamics), `doz` (Dozer)
- Format: `[prefix][number]` - e.g., `emp1`, `bat3`
- Keep IDs stable across updates

### Duration Rules

- Sprint = 14d (2 weeks)
- Use actual estimates from feature docs
- Always confirm with user if estimating

### Gantt Scope

- Show features/boulders, not tasks
- Include all milestones as diamond markers
- Max ~30 bars for readability
- Consolidate TBD pods into placeholder bars

## Output Format

```
## Roadmap Update Summary

**Updated**: [Pod Name(s)]

**Changes**:
- [Feature] status changed to [new status]
- [Feature] estimate revised from X to Y sprints
- [Feature] added to [Milestone]

**Target Impact** (vs product_targets.md):
- M&Ms: All must-haves covered | MISSING: [feature]
- [Milestone]: [status]

**Capacity Impact** (vs capacity.md):
- [Pod] has X eng-sprints available, Y committed
- [Flag if overallocated]

**Validation Impact**:
- [SHQ] may be affected by [change]

**Recommended Actions**:
1. [Action]
```

Then show the updated Gantt inline for review before committing.

## Notes

- Pod plans are the source of truth for priorities; roadmap.md is the consolidated view
- Milestone definitions and targets live in `product_targets.md`, not roadmap.md
- When adding features, always check if a `features/*.md` doc exists
- The Gantt is a communication tool - optimize for clarity, not completeness
