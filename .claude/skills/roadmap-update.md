# Roadmap Update Skill

You are helping update the project's **Feature Roadmap** - both the feature plans in `planning/pods/*_Plan.md` and the consolidated view in `generated/roadmap.md`.

> **Note**: This skill handles the Feature Roadmap (what features/boulders we're building, in which milestones, and when).
> Sprint-level task execution lives in **ClickUp**, not here.
> For Validation Roadmap updates (hypotheses, BHQs, SHQs), see the validation-review skill.
> For exploring alternative roadmap scenarios, see the roadmap-options skill.

---

## Project Structure

### Source of Truth (authored by humans)
- `planning/product_targets.md` - Milestone definitions, must-have features, success criteria (stable, rarely changes)
- `planning/pods/*_Plan.md` - Per-pod feature priorities by milestone (updated by pod leads)
- `planning/features/*.md` - Feature specs with estimates, discipline breakdown, implementation approach
- `planning/capacity.md` - Team staffing by discipline and milestone

### Generated / Consolidated
- `generated/roadmap.md` - **Consolidated Gantt + per-pod summaries** (regenerated by this skill from pod plans)

### Reference
- `planning/dependency_map.md` - Pod and feature dependencies
- `planning/GlobalRules.md` - Cross-project constraints
- `planning/ValidationRoadmap.md` - Validation hypotheses and SHQs

---

## Your Task

### 1. Read Current State
- Read the relevant `planning/pods/*_Plan.md` file(s) for the pod(s) being updated
- Read `planning/product_targets.md` for milestone definitions and targets
- Read `generated/roadmap.md` for the current consolidated view and Gantt
- Read `planning/capacity.md` to check staffing feasibility
- Read `planning/dependency_map.md` for constraints
- If cross-pod impact is possible, read all pod plans
- Read relevant `planning/features/*.md` if estimates or approach are changing

### 2. Gather Updates
Ask the user for:
- Which pod(s) to update?
- Feature status changes (started, completed, blocked)?
- New features to add?
- Priority or timeline changes?
- Features to move between milestones?
- Estimate revisions?

### 3. Validate Changes
- **Check capacity**: Does `planning/capacity.md` support the planned work? Flag overallocation.
- **Check dependencies**: Will changes break dependent pods or features?
- **Check targets**: Do changes affect must-have features in `planning/product_targets.md`? Flag if a must-have is being cut or delayed.
- **Check validation**: Do changes affect SHQs in `planning/ValidationRoadmap.md`? (mention if so, but don't update it here)
- Flag risks if you spot them

### 4. Update Pod Plan
- Update the relevant `planning/pods/*_Plan.md` file(s) with new priority order, statuses, estimates
- Update "Last Updated" timestamps
- If a new feature is added, note that a `planning/features/*.md` doc should be created (or create it)

### 5. Update Feature Docs (if needed)
- If estimates changed, update the feature doc's estimate and implementation flow
- If scope changed, update the feature doc
- If discipline needs changed, update the feature doc and flag in `planning/capacity.md`

### 6. Regenerate Consolidated Roadmap
**Always regenerate `generated/roadmap.md` after updating pod plans.**

#### Structure of `generated/roadmap.md`

The file follows this order:

1. **Header** - title, last updated, reference links, "do not manually edit" note
2. **High-Level Gantt** - Mermaid Gantt chart with all pods and milestones
3. **Pod Sections** - One section per pod, each containing:
   - Pod lead, producer, link to plan file
   - M&Ms (next milestone) validation focus: relevant BHQs/SHQs with status
   - Feature table for the current/next milestone
4. **Legend** - Gantt visual guide
5. **Update History** - Change log

#### a. Rebuild the Gantt Chart
Follow the Visual Roadmap Rules below. The Gantt is the first content section.

#### b. Rebuild Pod Sections
For each pod, generate a section with this structure:

```markdown
### [Pod Name]

**Pod Lead**: [Name] | **Producer**: [Name]
**Plan**: [`planning/pods/[Pod]_Plan.md`](../planning/pods/[Pod]_Plan.md)

**[Next Milestone] Validation Focus**: [1-sentence summary of what this pod is validating]

| Key BHQ | Key SHQs for [Milestone] | Status |
|---------|--------------------------|--------|
| [BHQ ID]: [Short name] | [SHQ IDs and brief descriptions] | [Status] |

**[Next Milestone] Features**:

| # | Feature | Estimate | Status |
|---|---------|----------|--------|
| 1 | [Feature] | X sprints | [Status] |
```

Pull validation info from the pod's plan file (Validation Focus / Validation Alignment sections).
Pull feature list from the pod's milestone priority table.
If the pod plan has no features defined yet, show `[TBD - awaiting feature definitions]`.

#### c. Target Check
After regenerating, verify planned features cover the must-haves from `planning/product_targets.md`.
If there are gaps, add a note in the Update History.

### 7. Summary
- Summarize what changed
- Highlight risks, dependency impacts, or target gaps
- If changes affect Validation Roadmap SHQs, note which ones
- Suggest next steps

## Visual Roadmap Rules

### Mermaid Gantt Syntax

```
gantt
    title [Chart Title]
    dateFormat YYYY-MM-DD
    axisFormat %b '%y
    tickInterval 4week

    section Milestones
    [Name]  :milestone, [id], [date], 0d

    section [Pod Name]
    [Feature Name]  :[status], [id], [start or dependency], [duration]
```

### Status Mapping

| Pod Plan Status | Gantt Status | Visual |
|----------------|-------------|--------|
| Completed | `done` | Gray bar |
| In Progress | `active` | Blue bar |
| Blocked | `crit` | Red bar |
| Not Started | (no tag) | Default bar |

### Task ID Rules

- Pod prefixes: `emp` (Empire), `meta` (Metagame), `bat` (Battle), `soc` (Social Dynamics), `doz` (Dozer)
- Format: `[prefix][number]` - e.g., `emp1`, `bat3`
- Keep IDs stable across updates

### Duration Rules

- Sprint = 14d (2 weeks)
- Use actual estimates from feature docs
- Always confirm with user if estimating

### Gantt Scope

- Show features/boulders, not tasks
- Include all milestones as diamond markers
- Max ~30 bars for readability
- Consolidate TBD pods into placeholder bars

## Output Format

```
## Roadmap Update Summary

**Updated**: [Pod Name(s)]

**Changes**:
- [Feature] status changed to [new status]
- [Feature] estimate revised from X to Y sprints
- [Feature] added to [Milestone]

**Target Impact** (vs product_targets.md):
- M&Ms: All must-haves covered | MISSING: [feature]
- [Milestone]: [status]

**Capacity Impact** (vs capacity.md):
- [Pod] has X eng-sprints available, Y committed
- [Flag if overallocated]

**Validation Impact**:
- [SHQ] may be affected by [change]

**Recommended Actions**:
1. [Action]
```

Then show the updated Gantt inline for review before committing.

## Notes

- Pod plans are the source of truth for priorities; `generated/roadmap.md` is the consolidated view
- Milestone definitions and targets live in `planning/product_targets.md`, not `generated/roadmap.md`
- When adding features, always check if a `planning/features/*.md` doc exists
- The Gantt is a communication tool - optimize for clarity, not completeness
