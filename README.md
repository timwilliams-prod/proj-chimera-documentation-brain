# Lotus Documentation Brain

A structured documentation system for managing game development across multiple pods, designed to work seamlessly with LLMs and version control.

## Purpose

This repo is the "brain" for project knowledge - a curated set of markdown files that:
- Provide LLMs (Claude, etc.) with exact context they need
- Serve as single source of truth for high-level planning
- Track dependencies and risks across pods
- Enable git-tracked, reviewable changes to roadmaps and priorities
- Separate **what we're building** (Feature Roadmap) from **why we're building it** (Validation Roadmap)

Data flows **from** Notion (design docs) and ClickUp (tasks) **into** these MD files, which are then used by LLMs and humans for fast, context-aware work.

## Core Files

### GlobalRules.md
- **Purpose**: Shared constraints (technical, process, business)
- **Update When**: Architecture decisions, policy changes
- **Used By**: All LLM tasks to understand boundaries

### pods/*_Backlog.md (one per pod)
- **Purpose**: Current/next sprint status for a single pod
- **Update When**: Sprint planning, mid-sprint progress, blockers emerge
- **Used By**: Roadmap updates, capacity planning, dependency tracking
- **Pods**: Empire, Metagame, Battle, Social Dynamics, Dozer

### ValidationRoadmap.md
- **Purpose**: Product validation tracking - are we building the right thing?
- **Structure**: 3x Winning Hypotheses → Big Hairy Questions (BHQs) → Small Hairy Questions (SHQs, 3-5 per milestone)
- **Update When**: Sprint evaluations, milestone reviews, new data/findings
- **Used By**: Validation reviews, risk evaluation, strategic planning

### roadmap.md (Feature Roadmap)
- **Purpose**: Visual Gantt chart of what we're building and when
- **Update When**: Auto-regenerated by `/roadmap-update` skill
- **Used By**: Sprint planning, stakeholder updates, cross-pod coordination

### dependency_map.md
- **Purpose**: Map of pod dependencies and shared resources
- **Update When**: New integrations, shared resource conflicts
- **Used By**: Impact analysis, risk evaluation, planning

### features/*.md
- **Purpose**: Detailed feature specs (goals, requirements, architecture, risks)
- **Update When**: Feature design, architecture changes, risk discovery
- **Used By**: Deep technical work, implementation, risk evaluation

## Skills

Skills are reusable prompts that guide Claude through specific workflows.

### `/roadmap-update`
Updates pod backlogs and regenerates the Feature Roadmap Gantt chart.

**Use When**:
- Sprint planning meetings
- Mid-sprint updates
- Reprioritization needed
- Blockers emerge

**What It Does**:
1. Reads the relevant pod backlog(s)
2. Asks for specific changes
3. Validates against dependencies and capacity
4. Updates pod backlog file(s)
5. Regenerates Mermaid Gantt in `roadmap.md`
6. Flags if changes affect Validation Roadmap SHQs

### `/risk-evaluation`
Scans all documentation for risks across timeline, technical, dependency, and validation dimensions.

**Use When**:
- Sprint planning (catch overcommitment)
- Mid-sprint (catch emerging issues)
- Pre-milestone (ensure readiness)
- Major feature starts

**What It Does**:
1. Reads all pod backlogs, dependency map, feature docs, and validation roadmap
2. Identifies risk patterns (timeline, technical, dependency, validation gaps)
3. Assesses impact x probability
4. Checks validation-feature alignment
5. Recommends actions

### `/validation-review`
Evaluates and updates the Validation Roadmap.

**Use When**:
- Every sprint (confidence check)
- Milestone boundaries (full assessment)
- New data arrives (playtest results, analytics)

**What It Does**:
1. Reviews SHQ progress for current milestone
2. Rolls up to BHQ and Winning Hypothesis confidence
3. Checks alignment between validation goals and feature work
4. Logs sprint evaluation with confidence trends
5. Flags gaps (SHQs without feature work, features without validation)

## Workflows

### Update Sprint Progress (2-3x/week)

```
/roadmap-update
```

Claude asks which pod, what changed, validates dependencies, updates the pod's backlog file and the Gantt chart.

### Validation Check (every sprint)

```
/validation-review
```

Claude reviews SHQ progress, updates confidence levels, checks feature-validation alignment.

### Risk Check (weekly or before planning)

```
/risk-evaluation
```

Claude scans all docs and outputs a prioritized risk report including validation gaps.

### Feature Work (any time)

```
"Help me implement [task for Empire pod]"
```

Claude reads the relevant pod backlog, feature docs, dependency map, and GlobalRules for full context.

## Data Ingestion

### From Notion
Notion is source of truth for **design documentation**:
- Game design docs → `features/*.md`
- Architecture docs → `features/*.md` (architecture section)
- High-level goals → `GlobalRules.md` (business constraints)
- Validation hypotheses → `ValidationRoadmap.md`

### From ClickUp
ClickUp is source of truth for **tasks and progress**:
- Sprint tasks → `pods/*_Backlog.md` (per pod)
- Blockers → `pods/*_Backlog.md` (On Hold section)

**Strategy**: Start manual (skills handle formatting). Automate later. See `INGESTION_GUIDE.md`.

**Important**: These MD files are **curated snapshots**, not live mirrors. ClickUp links preserved for drill-down.

## Best Practices

### Keep It High-Level
- These docs are **summaries**, not detailed specs
- ClickUp/Notion have the details; MD files have the "need to know"
- If a pod backlog is >300 lines, it's probably too detailed

### Update Frequently
- Stale docs are worse than no docs
- Pod backlogs: 2-3x/week (5 min each)
- Validation Roadmap: every sprint evaluation
- Feature docs: when design changes
- Dependency map: when integrations change

### Git Workflow
- Commit changes with meaningful messages
- "Update Empire sprint 23 progress - EMP-156 60% complete"
- PR for major roadmap shifts or validation pivots

### Two Roadmap Discipline
- **Feature Roadmap** answers "what and when"
- **Validation Roadmap** answers "is this the right thing"
- Keep them connected (SHQs trace to feature work) but separate

## Getting Started

1. **Fill in your Winning Hypotheses** in `ValidationRoadmap.md`
2. **Populate one pod backlog** (whichever is most active)
3. **Map dependencies** in `dependency_map.md` with your actual pod relationships
4. **Create 1-2 feature docs** using `features/combat_system.md` as template
5. **Try the skills** - `/roadmap-update`, `/validation-review`, `/risk-evaluation`
6. **Commit to git** and iterate

See `QUICK_START.md` for a step-by-step walkthrough.
